# Duplicating KnowledgeForest into a different Supabase account

This bundle recreates the **structure** of the KnowledgeForest backend (tables,
RPCs, RLS, triggers, cron, edge functions) in a fresh Supabase project owned by
a **different account**. It does **not** copy user content (pointers, edges,
attributes, embeddings, document chunks, forest structure) — those are
regenerated by ingestion + the nightly forest compute.

The schema was dumped from the live source project `rkuyvzcxaoulhjiflrmp` because
the base tables and most RPCs exist only in that project's dashboard, not in the
repo's `supabase/migrations/`. Replaying the migrations alone would **not** build
a working database.

## Bundle contents

| File | Purpose |
|------|---------|
| `schema_dump.sql` | Extensions, enums, 22 tables, constraints, indexes, 24 functions, triggers, RLS + policies, grants. |
| `seed_config.sql` | Required reference rows (access classes, dedup config, the Kibo tenant, schema vocabulary) + demo user clearance/membership. |
| `nightly_forest_compute_cron.tmpl.sql` | `trigger_nightly_forest_compute()` + cron schedule. **Has placeholders** for the target URL + anon key. |
| `../functions/` | All **11** edge functions (the repo previously had only 6; the other 5 were pulled from the live project — see note below). |

> **Edge functions note.** The deployed source has 11 functions:
> `insert-pointer`, `query-knowledge`, `ingest-document`, `ingest-batch`,
> `ingest-calendar`, `demo-reset` (were already in the repo) plus
> `compute-forest`, `link-pointers`, `log-query-path`, `backfill-embeddings`,
> `backfill-vocab-embeddings` (added to `supabase/functions/` from the live
> project). `compute-forest` is required by the nightly cron.

## Prerequisites (local machine)

```bash
# macOS
brew install supabase/tap/supabase     # Supabase CLI (not installed in this env)
```

You need, from the **target** account:
- A Supabase access token (Dashboard > Account > Access Tokens) for `supabase login`.
- The OpenAI API key the functions should use (reuse the source's, or a new one).

## Steps (run against the TARGET account)

### 1. Create the project
Create a new project in the target account. Choose **Postgres 17** and a region.
Record from **Project Settings > API**:
- Project URL — `https://<NEW_REF>.supabase.co`
- `anon` (public) key
- `service_role` key

### 2. Apply the schema
In the target project's **SQL Editor**, run, in order:
1. `schema_dump.sql`
2. `seed_config.sql` **Part A** (Part B needs the demo user — step 5).

`schema_dump.sql` creates the extensions itself (`vector`, `pg_trgm`, `pg_cron`,
`pg_net`, `pgcrypto`, `uuid-ossp`, `moddatetime`). If your dashboard restricts
`create extension`, enable those under **Database > Extensions** first.

### 3. Deploy edge functions
```bash
supabase login                              # paste target access token
supabase link --project-ref <NEW_REF>
supabase functions deploy                   # deploys all functions in supabase/functions/
```
All functions read `SUPABASE_URL`, `SUPABASE_ANON_KEY`,
`SUPABASE_SERVICE_ROLE_KEY` from the platform automatically — no edits needed.

### 4. Set the OpenAI secret
```bash
supabase secrets set OPENAI_API_KEY=sk-...  # required for embeddings + LLM planning
```

### 5. Create the demo auth user (fixed UID)
The app signs in as `partner@kibo.demo` / `kibo-partner`, and the seed grants are
keyed to that user's **fixed** id `aaaaaaaa-0000-4000-8000-aaaaaaaaaaaa`. Create
the user with that exact id. Easiest via the Admin API (service_role key):

```bash
curl -X POST 'https://<NEW_REF>.supabase.co/auth/v1/admin/users' \
  -H "apikey: <SERVICE_ROLE_KEY>" \
  -H "Authorization: Bearer <SERVICE_ROLE_KEY>" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "aaaaaaaa-0000-4000-8000-aaaaaaaaaaaa",
    "email": "partner@kibo.demo",
    "password": "kibo-partner",
    "email_confirm": true,
    "user_metadata": { "name": "Kibo Partner" }
  }'
```
Then run **Part B** of `seed_config.sql` (tenant membership + class grants).

### 6. Schedule the nightly cron
Edit `nightly_forest_compute_cron.tmpl.sql`, replacing `__TARGET_PROJECT_URL__`
and `__TARGET_ANON_KEY__` with the target values from step 1, then run it in the
SQL Editor.

### 7. (Optional) Backfill vocabulary embeddings
`seed_config.sql` inserts schema vocabulary with `NULL` embeddings — regex query
hints work without them, but semantic hints don't. To enable semantic hints:
```bash
curl -X POST 'https://<NEW_REF>.supabase.co/functions/v1/backfill-vocab-embeddings' \
  -H "Authorization: Bearer <ANON_OR_SERVICE_KEY>"
```

### 8. Repoint the app
Update local config (no source-code changes — the clients are env-driven):

`.env.local`
```
VITE_SUPABASE_URL=https://<NEW_REF>.supabase.co
VITE_SUPABASE_ANON_KEY=<target anon key>
# VITE_KIBO_TENANT_ID stays ca61f0e5-563e-5894-954f-38f5a9e0eabc (tenant id preserved)
```

`pipeline/.env`
```
SUPABASE_URL=https://<NEW_REF>.supabase.co
SUPABASE_SERVICE_ROLE_KEY=<target service_role key>
```

## Verification

1. **Structure** — in the target SQL Editor:
   ```sql
   select count(*) from information_schema.tables where table_schema='public';   -- 22
   select count(*) from pg_proc p join pg_namespace n on n.oid=p.pronamespace
     where n.nspname='public'
       and not exists (select 1 from pg_depend d where d.objid=p.oid and d.deptype='e'); -- 25 (24 + trigger_nightly_forest_compute)
   select jobname, schedule from cron.job;                                        -- nightly-forest-compute, 0 3 * * *
   ```
2. **Reference data**
   ```sql
   select key from access_classes;                  -- public, confidential, restricted
   select count(*) from schema_vocabulary;          -- 35
   ```
3. **Edge functions** — `insert-pointer` and `query-knowledge` return 200 (not
   404/500) when called with the target anon key.
4. **RLS** — anonymous reads return only public-class rows; the demo user (after
   sign-in) additionally sees confidential/restricted. Mirror
   `supabase/tests/access_class_rls_verification.sql`.
5. **End-to-end** — `npm run dev` with the new `.env.local`, sign in as the demo
   user, run the live-demo ingest + a knowledge search, confirm results render
   and the 3D forest draws.
